# User Management Preservation

This document preserves the current user-management logic, schema decisions, page structure, and workflow rules implemented in Project LINK so the feature can be migrated, refactored, or extended without losing important behavior.

It reflects the implemented code and docs in this repository, not older planning assumptions.

## Purpose

The admin users area is no longer a simple create/edit table. It is now a complete account-management workflow with:

1. A filterable user directory
2. A create-user flow with normalized profile fields
3. A manage-user flow for account state, scope, and notes
4. A temporary-password reset workflow
5. A skippable first-login password reminder
6. Optional private profile photos
7. Database and edge-function contracts that enforce admin-only updates

The main migration risk is rebuilding only the visible form and table while losing the less obvious rules around scope, password state, deactivation, email sync, and photo storage.

## Domain Model

### 1. `user_profiles` is now a normalized staff profile, not a loose auth mirror

The current user profile record is designed for admin operations, not just sign-in.

Preserved fields:

- `id`
- `username`
- `date_of_birth`
- `sex`
- `mobile_number`
- `role`
- `health_station_id`
- `purok_assignment`
- `is_active`
- `must_change_password`
- `created_at`
- `updated_at`

Added fields:

- `user_id`
- `first_name`
- `middle_name`
- `last_name`
- `name_suffix`
- `email`
- `alternate_mobile_number`
- `coverage_notes`
- `admin_notes`
- `last_login_at`
- `password_changed_at`
- `deactivation_reason`
- `profile_photo_path`
- `profile_photo_updated_at`
- `created_by`
- `updated_by`

Removed field:

- `full_name`

Important preservation rule:

- The display name is now composed from normalized fields, not stored as a single authoritative text field.

### 2. `user_id` is a human-readable system identifier

`user_id` is not the database primary key. It is an admin-facing identifier generated by the database.

Format:

- `USR-YYYY-####`

Current generation behavior:

- implemented in `generate_human_user_id()`
- assigned by the `assign_user_profile_defaults()` trigger
- uniqueness enforced by a unique index

Important preservation rule:

- `user_id` is for UI, admin operations, and traceability
- UUID `id` remains the actual relational key

### 3. Email is intentionally duplicated in Auth and `user_profiles`

The current design stores email in both places:

- `auth.users.email`
- `public.user_profiles.email`

Reason:

- admin tables and profile views need a queryable email without depending on Auth joins
- frontend user-management pages read email directly from `user_profiles`

Important preservation rule:

- changing email must update both Auth and `user_profiles`
- treating `user_profiles.email` as optional or display-only would break the current admin workflow model

### 4. User scope is driven by role and `health_station_id`

The current account model does not store a direct barangay ID on the user.

Scope is represented by:

- `role`
- `health_station_id`
- `purok_assignment` for BHW only

Important role rules:

- `bhw` and `midwife_rhm` require `health_station_id`
- `nurse_phn`, `dso`, `phis_coordinator`, `city_health_officer`, and `system_admin` are city-wide
- `purok_assignment` is shown for `bhw`
- `purok_assignment` is warning-only when blank

Important preservation rule:

- user scope is BHS-based, not directly barangay-based
- if barangay context is needed, derive it from the assigned health station and its related records

### Main form sections

Current form sections:

### Identity fields

Current identity fields:

- `first_name`
- `middle_name`
- `last_name`
- `name_suffix`
- `date_of_birth`
- `sex`

Rules:

- first name is required
- last name is required
- middle name is optional
- suffix is optional
- DOB is required
- sex is required and limited to `M` / `F`

### Contact fields

Current contact fields:

- `email`
- `mobile_number`
- `alternate_mobile_number`

Rules:

- email is required
- both phone fields use `+639XXXXXXXXX` format
- alternate mobile is optional

### Role and access fields

Current role/access fields:

- `user_id` display
- `username`
- `role`
- `initial_password` in create mode only
- `must_change_password` switch in edit mode
- `is_active` switch in edit mode
- `deactivation_reason` when inactive

Rules:

- `user_id` is generated and read-only
- username is suggested from first and last name, but editable
- create mode requires an initial password with minimum length 12
- edit mode exposes password reminder state and account active state
- inactive accounts require a deactivation reason

### Assignment and coverage fields

Current scope fields:

- `health_station_id`
- `purok_assignment`
- `coverage_notes`

Rules:

- `health_station_id` is required only for `bhw` and `midwife_rhm`
- city-wide roles disable station assignment
- `purok_assignment` is shown only for `bhw`
- `coverage_notes` is free text

Important preservation rule:

- scope fields are actively cleared when role changes make them invalid

### Admin notes

Current admin-only metadata field:

- `admin_notes`

Purpose:

- preserve private operational/admin context that should not be exposed as user-facing profile data

## Password Logic

### 1. Create-user password behavior

When an admin creates a user:

- they set a temporary initial password
- the profile is created with `must_change_password = true`

### 2. First-login reminder behavior

The first-login password dialog is currently skippable.

Actual current behavior:

- if `must_change_password = true`, the dialog appears
- user may change password now
- user may also choose `Skip for now`
- skipping only dismisses the dialog for the current session
- the reminder returns on a future login while the flag remains true

Important preservation rule:

- UI copy must never claim the password change is fully mandatory before continuing

### 3. Password completion behavior

When the user successfully changes their password:

- Supabase Auth password is updated
- `complete-password-change` clears `must_change_password`
- `password_changed_at` is recorded
- session is refreshed so claims update

### 4. Admin password reset behavior

Admin password reset is a separate workflow.

Behavior:

- admin sets a new temporary password
- password is updated in Auth through `reset-user-password`
- `must_change_password` is set back to `true`
- user is expected to see the reminder again later

Important preservation rule:

- password reset is not embedded as a field in the main manage-user form

## Profile Photo Logic

### Storage model

Current photo bucket:

- `user-profile-photos`

Path-building behavior:

- bucket path is based on the UUID `id`
- file name uses a random UUID plus the original extension

Reason:

- stable technical storage key
- avoids coupling storage identity to editable values like username

### Access model

Current photo access is private:

- authenticated users can read
- system admins can insert, update, and delete

### Rendering model

Photo rendering uses signed URLs plus initials fallback.

Important preservation rule:

- photo presence should never be required for identity display

## Backend Contract Decisions

### Edge-function ownership

Admin-sensitive writes are intentionally routed through edge functions, not direct client table updates.

Current functions:

- `create-user`
- `update-user`
- `reset-user-password`
- `complete-password-change`
- `record-last-login`

Important preservation rule:

- do not move sensitive admin profile writes back to direct client-side table updates without replacing the permission model

### `self_update` removal

The redesign removes `self_update` on `user_profiles`.

Meaning:

- user profile updates are no longer assumed to be safe for direct self-service writes
- admin-sensitive profile state is controlled through scoped server paths

## Validation And Warnings

### Hard validation

Current hard validation includes:

- required first name
- required last name
- required email
- required username
- required DOB
- required sex
- required role
- required station for `bhw` and `midwife_rhm`
- required deactivation reason when inactive
- mobile format validation
- alternate mobile format validation
- user ID format validation in edit mode
- minimum 12-character temporary password

### Soft warnings

Current warning logic includes:

- BHW without purok assignment
- Midwife without BHS assignment
- PHN with BHS assignment
- system admin with BHS assignment
- inactive user still flagged for password change
- field-heavy role without mobile number

Important preservation rule:

- warnings are advisory and should not silently become blockers unless the product deliberately changes policy

## Barangay / BHS Scope Notes

The user profile does not directly store a barangay ID.

Current scope model:

- user -> `health_station_id`
- health station -> operational health-station records and related barangay coverage

Implications:

- if migration targets a direct barangay-aware model, that is a schema redesign, not a no-op port
- current system logic is BHS-scoped first

## Migration Checklist

If this user-management feature is migrated or rebuilt, preserve the following:

- normalized name fields instead of `full_name`
- generated `user_id` separate from UUID `id`
- duplicated queryable `email` in `user_profiles`
- BHS-based user scoping
- city-wide roles staying unscoped
- skippable first-login password reminder
- separate admin password reset workflow
- deactivation reason requirement
- right-rail manage-user context cards
- profile-photo private storage model
- broad multi-field directory search
- mobile card presentation for the directory
- admin-only edge-function ownership of sensitive writes

## Main Risks If Rebuilt Incorrectly

- restoring a single `full_name` field would lose the normalization decision
- treating email as Auth-only would weaken admin queryability
- removing `user_id` would lose a human-readable operational identifier
- turning the password reminder into a hard block would change current behavior
- allowing direct self-updates again would weaken the security model
- replacing BHS scope with direct barangay scope would change the account model
- making profile photos public would change privacy behavior
- reducing the user list to a simple table would lose current filter and mobile behavior

## Short Summary

The preserved mental model is:

- `user_profiles` is now a richer staff-account record
- names are normalized
- `user_id` is human-readable but UUID remains the true key
- scope is BHS-first, not barangay-first
- password reminder is real but skippable
- admin password resets are separate from normal profile edits
- profile photos are optional and private
- the users area is an operational account-management workflow, not a lightweight CRUD page
